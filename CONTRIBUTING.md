# Contributing to Decision Assessment Assistant

First off, thank you for considering contributing to Decision Assessment Assistant! It's people like you that make this project such a great tool.

## Code of Conduct

This project and everyone participating in it is governed by the [Decision Assessment Assistant Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code. Please report unacceptable behavior to **community@example.com**.

## How Can I Contribute?

### Reporting Bugs

This is one of the most helpful ways you can contribute. Before creating a bug report, please check the [existing issues](https://github.com/hamadalmunyif/decision-assessment-assistant-docs/issues) to see if your problem has already been reported.

If you can't find an open issue addressing the problem, [open a new one](https://github.com/hamadalmunyif/decision-assessment-assistant-docs/issues/new). Be sure to include a **title and clear description**, as much relevant information as possible, and a **code sample or an executable test case** demonstrating the expected behavior that is not occurring.

### Suggesting Enhancements

If you have an idea for an enhancement, we'd love to hear it! Please [open an issue](https://github.com/hamadalmunyif/decision-assessment-assistant-docs/issues/new) and describe your suggestion in detail. The more information you can provide, the better.

### Your First Code Contribution

Unsure where to begin contributing? You can start by looking through these `good first issue` and `help wanted` issues:

- [Good first issues](https://github.com/hamadalmunyif/decision-assessment-assistant-docs/labels/good%20first%20issue) - issues which should only require a few lines of code, and a test or two.
- [Help wanted issues](https://github.com/hamadalmunyif/decision-assessment-assistant-docs/labels/help%20wanted) - issues which should be a bit more involved than `good first issue` issues.

### Pull Requests

1. Fork the repo and create your branch from `master`.
2. If you've added code that should be tested, add tests.
3. If you've changed APIs, update the documentation.
4. Ensure the test suite passes.
5. Make sure your code lints.
6. Issue that pull request!

## Styleguides

### Git Commit Messages

- Use the present tense ("Add feature" not "Added feature").
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...").
- Limit the first line to 72 characters or less.
- Reference issues and pull requests liberally after the first line.

### Documentation Styleguide

- Use Markdown.
- Try to stick to the existing style.
- Use a consistent voice and tone.

## Any contributions you make will be under the CC BY 4.0 Software License

In short, when you submit code changes, your submissions are understood to be under the same [CC BY 4.0](https://creativecommons.org/licenses/by/4.0/) that covers the project. Feel free to contact the maintainers if that's a concern.
